# AGENTS.md

## Project
Grow Simulator is a mobile-first cannabis grow simulation game.
The codebase uses a deterministic simulation loop with persistent state.
All event logic must feel realistic, phase-appropriate, and consistent with plant status and climate conditions.

## Core Goal
Rebuild the event system into a modular, simulation-driven architecture.

## Non-Negotiable Rules
- Do not implement random flavor events without system grounding.
- Every player-facing event must be phase-valid and condition-valid.
- Negative events must derive from plant phase, climate, stress, risk, root-zone state, and recent history.
- Positive reward events must derive from sustained stability or good recovery.
- Preserve compatibility with existing stress and risk systems.
- Do not invent fake plant science. Prefer generic, realistic causality.
- Avoid monolithic rewrites. Prefer incremental modular refactors.
- Do not break current boot flow, save/load behavior, or sim-time continuity.
- Keep the app mobile-first and premium in UI behavior.

## Architecture Targets
Create these modules under src/events:
- eventDefinitions.js
- eventEvaluator.js
- eventRuntime.js
- eventConsequences.js
- eventRewards.js
- eventAnalysis.js
- eventUtils.js
- eventTypes.js

## Runtime Requirements
- Use sim-time, not wall-clock timing
- Support active events, warnings, escalations, resolutions, reward events, and event chains
- Support hidden internal risk buildup that does not always create a popup immediately
- Support event cooldowns and anti-spam behavior
- Support event history and post-event analysis

## Reward Rules
- Use existing stress and risk values
- Add stableTimeMs and rewardCooldownMs
- Trigger rewards only after sustained good control or successful recovery
- Rewards may suppress negative event pressure temporarily

## UI Requirements
- Event popup for immediate decisions
- Central event screen for active, past, and analyzed events
- Event detail screen with cause, action taken, effect, and learning outcome

## Code Quality
- Small focused functions
- No duplicated event logic
- Avoid hidden side effects
- Add lightweight tests where possible
- Keep naming explicit and consistent
